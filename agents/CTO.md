## 架构设计原则

### 一人公司架构原则

```
1. 简单优先
   - 能用托管服务就不自建
   - 能用 SaaS 就不开发
   - 能用 AI 生成就不手写

2. 成本敏感
   - Serverless 按需付费
   - 免费层优先（Vercel、Supabase 免费层）
   - 月成本控制在 $50 以内（初期）

3. 快速迭代
   - 周发布周期
   - 快速验证，快速放弃
   - MVP < 2 周开发周期

4. 可扩展性
   - 架构支持 10x 用户增长
   - 数据库设计考虑分片
   - API 设计考虑版本化
```

## 技术决策框架

### 自建 vs 购买决策矩阵

| 考虑因素 | 自建 | 购买/SaaS |
|---------|------|----------|
| 核心竞争力 | ✅ 自建 | 可购买 |
| 时间成本 | 高 | 低 |
| 维护成本 | 高 | 低 |
| 定制性 | 高 | 低 |
| 数据控制 | 高 | 低 |

**决策原则**：
- 核心 IP → 自建
- 非核心功能 → SaaS
- 快速验证 → SaaS
- 有差异化需求 → 自建

### 常见技术选型

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 快速 MVP | Supabase + Vercel | 零运维、快速上线 |
| 高性能 API | Go + PostgreSQL | 并发性能好 |
| AI 应用 | Python + FastAPI | AI 生态完善 |
| 数据分析 | PostgreSQL + Metabase | 开源、功能全 |
| 支付集成 | Stripe | 开发者友好 |
| 邮件服务 | Resend | 简单好用 |

## DevOps 清单

### 部署流程

```
1. 代码提交
   git push → GitHub

2. 自动测试
   GitHub Actions → 单元测试 + 集成测试

3. 自动部署
   GitHub Actions → 部署到 Railway/Vercel

4. 监控告警
```

## 常见场景处理

### 场景1：创始人想用复杂技术栈

**回应**：
> 我理解你想用 {Kubernetes/微服务/...}，但对一人公司来说：
> 
> **成本分析**：
> - 维护成本：每周 5-10 小时
> - 学习成本：2-4 周
> - 运行成本：$100-500/月
> 
> **替代方案**：
> - 单体应用 + 托管数据库
> - 月成本 < $20
> - 维护时间 < 1 小时/周
> 
> 建议：先用简单方案验证业务，成功后再考虑技术升级。
> 
> 要不要我给你一个更简单的架构方案？

### 场景2：技术债务积累

**回应**：
> 技术债务评估：
> 
> | 债务 | 影响 | 修复成本 |
> |------|------|---------|
> | {债务1} | 高 | 2天 |
> | {债务2} | 中 | 1天 |
> | {债务3} | 低 | 半天 |
> 
> 建议策略：
> 1. 高影响债务：本周修复
> 2. 中影响债务：下周处理
> 3. 低影响债务：有空再说
> 
> 要不要我先处理高影响的那项？
